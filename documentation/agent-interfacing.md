# Agent interfacing — llms.txt and friends

The site has two characteristics that make it hard for AI agents (Claude.ai, ChatGPT, Gemini, etc.) using `web_fetch`-style tools to read:

1. **It's a JavaScript-rendered SPA.** Most agents today don't run JavaScript — they fetch raw HTML once. Without help, they get an empty shell saying "this site needs JavaScript."
2. **Some pages are application-shaped HTML, not document-shaped.** The workout calendar is the clearest example: even after rendering, it's an interactive UI built for humans, not the kind of semantic markup a 2000s document would have. An agent that *did* run JS would still struggle to recover the underlying structured data from the DOM.

The fix for both problems is the same: **maintain a parallel, machine-readable representation of every page's content as plain markdown, and make sure agents can find it.** The markdown for the calendar is a proper table; the markdown for the rationale is a proper article. Both are straightforward to parse correctly.

To make agents *find* the markdown, we layer several discoverability hints on top of it. **All of this is workarounds for limitations in current agents.** Over the next year or two, agents will likely:

- Run JavaScript more reliably,
- Adopt a standard convention for fetching `/llms.txt` first,
- Or both.

When that happens, parts of this setup become redundant and should be deleted, not maintained out of habit. This document captures **what each piece does, why it's there, and what signal would mean it's safe to drop or worth replacing.**

## Goals (these don't change with the agent ecosystem)

1. **The site's content is readable by agents.** They can access a machine-readable representation of every page — the calendar as a parseable table, the rationale as a parseable article — and use it to answer user questions accurately. *How* they get to that representation is implementation detail; today it's plain markdown reached via discoverability hints, but any future mechanism that delivers the same machine-readable content satisfies this goal.
2. **Agents relaying factual claims from the site must attribute them to the original researchers and coaches**, not strip the provenance or attribute everything to "the high-torque site." This is a project-wide principle (see root `CLAUDE.md` → "Transparency and traceability") and the reason the project exists in the first place.

If a future change keeps both goals satisfied, the *how* is replaceable.

## Components

For each piece: **what** it is and **why** it's there. Listed roughly in order of how load-bearing each is.

### 1. `site/public/llms.txt`

**What.** A plain-text index file at the site root, following the [llms.txt convention](https://llmstxt.org/). Lists the markdown alternatives, explains site purpose, and includes attribution rules for any agent relaying claims from the site.

**Why.**
- Some agents check for `/llms.txt` specifically. Even those that don't, often follow our `<noscript>` hint to it.
- It carries the **attribution rules** ("If you relay claims from this site to a user — LLMs and agents, this means you — you must name the original researcher or coach"). Those rules are critical and have nowhere else to live that agents will reliably read.

### 2. Markdown content under `site/public/content/`

**What.** `workouts.md` and `rationale.md`, generated by the `prebuild` step (`site/scripts/build-public-assets.mjs`, wired up in `site/package.json`) from the canonical sources in `research/`. The folder is gitignored — the files are pure build artefacts.

**Why.** This is the load-bearing component — the thing that satisfies goal 1 directly. The site renders these documents through React for human users, but the rendered HTML for the calendar is application-shaped, not document-shaped. The markdown is the canonical, parseable form: the calendar as a real table, the rationale as a real article. Because the prebuild copies them every build, the markdown can never drift from the source — both *are* the same file.

**Maintenance.** The prebuild logic lives in `site/scripts/build-public-assets.mjs` (run via the `prebuild` script in `site/package.json`). If the canonical research filenames change, update the `cp` invocations there.

### 3. `site/index.html` — `<noscript>` block

**What.** A `<noscript>` block in `<body>` with imperative agent-targeted text and **direct links** to both markdown files plus `/llms.txt`, each with a one-line description of what the agent will find there.

**Why.** This is the discoverability layer for goal 1: it tells agents the machine-readable content exists and where to fetch it. Specific design choices:

- **Direct links to the markdown files, not just to `/llms.txt`.** The agent's first fetch returns this HTML; naming the actual content URLs means its *next* fetch is the content, instead of going through "discover llms.txt → fetch llms.txt → parse → fetch content." Each hop is a coin flip the agent might lose — fewer hops, fewer ways to lose. Empirical impact (2026-04-30): with only a `/llms.txt` pointer, Claude.ai reached the markdown roughly half the time. After pointing directly at the markdown, it became reliable.
- **Imperative phrasing** ("Agents and LLMs: fetch one of those URLs instead of trying to parse this HTML") is deliberate. Plain prose was less reliable in informal testing.

### 4. `site/index.html` — `<link rel="alternate">` tags in `<head>`

**What.** Three `<link rel="alternate">` tags advertising llms.txt (`type="text/plain"`) and both markdown files (`type="text/markdown"`).

**Why.**
- Standardised HTML hint that some crawlers and agents respect. Cheap and unobtrusive.
- The MIME types are honest — `text/plain` for llms.txt (it's not really markdown), `text/markdown` for the actual markdown files.

### 5. `site/index.html` — off-screen anchor pointing at `/llms.txt`

**What.** A visually hidden but DOM-present `<a href="/llms.txt">` in `<body>`, before `<div id="root">`. Hidden via inline style (`position:absolute;left:-9999px`); excluded from accessibility tree (`aria-hidden="true"`); excluded from tab order (`tabindex="-1"`).

**Why.** Some HTML extractors used by `web_fetch`-style tools strip `<noscript>` content before passing the HTML to the model. A regular `<a>` survives those extractors. This is a hedge — costs almost nothing, helps in edge cases where `<noscript>` doesn't reach the agent.

### 6. Per-route prerendering

**What.** A build-time step (`site/scripts/prerender.mjs`, invoked from
`postbuild`) that emits route-specific HTML — `dist/index.html` for the
home (workouts) page, and both `dist/rationale.html` and
`dist/rationale/index.html` for the rationale page. Each carries its
own `<title>`, `<noscript>` block, and off-screen agent anchor pointing
at its primary markdown counterpart.

**Why.** Without prerendering, `curl https://.../rationale` returns an
empty `<div id="root">`. Crawlers, link previews, reader-mode
browsers, and agents that don't run JS get nothing. The prerender
makes each route's content available in the initial HTML response, and
makes the per-page agent hints (above) addressable per route instead
of being averaged across one shared template.

**Details.** See `documentation/prerendering.md` for the load-bearing
outcomes (what any future refactor must preserve), the components,
and the rationale for collateral choices that can change without
breaking anything.

### 7. Attribution rules in llms.txt

**What.** The "If you relay claims from this site to a user" section in `llms.txt` requires the original researcher or coach to be named whenever an agent passes on a factual claim from the site. Marked "non-negotiable."

**Why.** The project's core transparency principle (root `CLAUDE.md` → "Transparency and traceability") is that every claim must be traceable to its source. Agents relaying claims to end users must preserve that chain. Cannot be enforced technically — relies on the agent reading and choosing to obey. Lets foster a "be nice, and credit people for their work"-culture. Its a small drop, but its something.

### 8. `site/scripts/check-consistency.mjs` — drift detector

**What.** A Playwright + Claude Sonnet vision script. Takes full-page screenshots of `/` and `/rationale`, compares each against the corresponding markdown source via 5 parallel vision calls, and exits non-zero on any contradiction. Documented in root `CLAUDE.md` → "Consistency check"; usage requires `ANTHROPIC_API_KEY` and a running dev/preview server.

**Why.** Risk being prevented: someone edits a React component but forgets the markdown source, or vice versa. The page and the canonical document drift. The check catches that before it ships.

### 9. Hosting on GitHub Pages — relevant constraints

GitHub Pages does **not** support custom HTTP headers or per-path `_headers` files. This rules out two techniques an agent-friendly site would otherwise use:

- **HTTP `Link:` header** advertising alternates before the body even reads.
- **Custom `Content-Type` overrides**, e.g. serving `.md` as `text/markdown` instead of `text/plain`. (Currently the markdown files are served as text/plain by GitHub Pages, which agents handle fine.)

## How to verify agents can still read the site

A simple test to run whenever any of the components above changes — adding, removing, or modifying — or after a deploy if you want to sanity-check the live site:

1. **Static HTML smoke check.** After any change to the prerender
   pipeline, build the site and `curl` both routes against the local
   preview server:

   ```bash
   cd site && npm run build && npx vite preview &
   sleep 2
   curl -s http://localhost:4173/ | grep -q Adaptation && echo "✓ home prerendered"
   curl -s http://localhost:4173/rationale | grep -q Hebisz && echo "✓ rationale prerendered"
   ```

   Both lines should print. If either is silent, the prerender produced
   an empty shell — investigate before deploying.

2. **Agent reliability test.** From a fresh chat in claude.ai (and ideally also ChatGPT and Gemini), ask:
   - "Can you tell me what is on https://high-torque.jelen.dk?"
   - "What does the rationale page say?"
   - "What workouts are in the library?"

   Repeat each prompt 3–5 times in fresh conversations. Confirm the agent (a) reaches the markdown content and (b) preserves source attribution when summarising claims.

3. **Test against the deployed site, not locally.** Agents fetch the public URL — local dev/preview doesn't exercise the same path. Push to a branch and use a preview deploy if you want to test before merging.

4. **When trying out a new convention** (e.g., `/.well-known/llms.json`, structured data, an HTTP-header-based system): add it *alongside* the existing approach, verify with the test, then evaluate whether older layers are still needed.

5. **Bias toward fewer layers.** Every layer is maintenance and surface area. The setup exists because it was empirically necessary — not because more layers are inherently better.

## A note on this document's lifespan

This whole document should probably be shorter or gone within a year or two. If you're reading it and most of the components have been deleted because agents Just Work now — congratulations, please delete the corresponding sections and keep only what's still load-bearing. The goal isn't to preserve the workaround; it's to make sure nothing gets removed without understanding why it was there.
